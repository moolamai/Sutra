# Lawyer: workflows

## Task graphs

Matter preparation decomposes into prerequisite graphs the same way a syllabus does. A litigation graph, abbreviated:

```
facts.intake -> issues.spotting -> authority.research -> theory.selection -> drafting.pleadings -> hearing.preparation
                       |                    |
                       +--> limitation.check (prerequisite of everything downstream)
                       +--> jurisdiction.confirmation
```

Weak foundations trigger loop-back exactly as in every domain: a drafting session that hits friction because the authority research is thin routes backwards to `authority.research` before continuing.

## Guidance mode mapping

| Mode | In this domain |
|---|---|
| `exploratory` | Issue-spotting dialogue: the companion asks what outcomes the client needs, surfaces angles |
| `guided` | Clause-by-clause contract walkthrough against the playbook |
| `reinforcement` | Matter refresh before a hearing: re-surface the chronology, key authorities, weak points |
| `prerequisite-remediation` | Loop back to research or facts when drafting friction reveals gaps |
| `diagnostic` | New matter intake: calibrate what is known, what is assumed, what is missing |

## The canonical matter loop

1. The lawyer works in a matter session; friction signals (long hesitation over a clause, repeated rephrasing of a query) accrue like any other subject evidence.
2. The router holds, loops back, or advances through the matter graph; the rationale string is the file note.
3. Reasoning runs with counterargument steps mandatory for theory-of-the-case work.
4. Drafts persist through the `write` tools; filings stop at the `critical` gate for human sign-off.
5. Matter memory accumulates corrections and milestones that survive to appeal, transfer, or archive.
