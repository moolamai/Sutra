# Package Publish Checklist

Release operators use this checklist before any P5/P7 publish rehearsal.

## Scope

- Public npm packages in this monorepo are `@moolam/*` scoped packages and unscoped `sutra-sdk` / `sutra-bindings-*` names under `packages/` with `private` not set to `true`.
- Production publish is locked until Protocol 1.0 freeze RFC acceptance. The mechanical gate is `pnpm production-publish:gate` → [`rfcs/appendix/production-publish-gate.json`](../../rfcs/appendix/production-publish-gate.json), driven by [`rfcs/0001-protocol-1.0-freeze.md`](../../rfcs/0001-protocol-1.0-freeze.md).
- **Current state (2026-07-19):** gate unlocked; local `publish:readiness` and `publish:pack` green. Production npm publish pending `v1.0.0` tag through `.github/workflows/release.yml` (operator secrets and variables on the repo — not committed).

## Required gates (repo root)

Run these commands in order:

1. `pnpm publish:readiness`
2. `pnpm publish:pack`
3. `pnpm publish:readiness:prove`
4. `pnpm publish:pack:prove`

If any step fails, do not continue to release workflow tasks.

## Per-package dry-run command

For targeted investigation, run:

`pnpm --filter <package-name> pack --dry-run`

Example:

`pnpm --filter sutra-sdk pack --dry-run`

## npm provenance (CI publish attestation)

Provenance is generated by `.github/workflows/release.yml` in CI — never post-hoc from a laptop.

### Required GitHub repository settings

1. **Actions workflow permissions** — Settings → Actions → General → Workflow permissions: **Read and write** (or fine-grained equivalent allowing `id-token` issuance).
2. **OIDC trust for npm** — `permissions.id-token: write` is declared in `release.yml`; GitHub Actions must be allowed to mint OIDC tokens for publish jobs.
3. **Secrets and variables**
   - `NPM_TOKEN` secret — registry auth token for publish (GitHub Packages scratch and/or npmjs.com production).
   - `NPM_SCRATCH_REGISTRY_URL` variable — scratch scope default `https://npm.pkg.github.com`.
   - `NPM_PROD_REGISTRY_URL` variable — production default `https://registry.npmjs.org` (P7 gate only).
   - `NPM_ALLOW_PROD_PUBLISH` variable — must remain `false` until the freeze RFC is Accepted and `pnpm production-publish:gate` reports `unlocked: true` (`FREEZE_RFC_UNLOCKED`). Setting the var early does nothing: `release.yml` refuses production registries while the gate is locked.
   - `PYPI_ALLOW_PROD_PUBLISH` variable — same freeze unlock requirement for production PyPI.
4. **npm trusted publishing** — on [npmjs.com](https://www.npmjs.com), link each `@moolam/*` package to this GitHub repository and enable **trusted publishing** / provenance for GitHub Actions before any production tag.

### When provenance is enabled

- **Production registry** (`registry.npmjs.org`) with `NPM_ALLOW_PROD_PUBLISH=true` **and** `FREEZE_RFC_UNLOCKED=true`: `pnpm changeset:publish` runs via `scripts/run-changeset-publish.mjs` with `NPM_CONFIG_PROVENANCE=true` and OIDC credentials from `actions/setup-node`.
- **Scratch rehearsals** (`v*-rehearsal.*`, `v*-rc.*` on GitHub Packages): publish runs without provenance (`PROVENANCE_ENABLED=false`) — rehearsal validates install path only.

## Protocol 1.0 freeze → production unlock

1. Follow the Maintainer acceptance workflow in [`rfcs/0001-protocol-1.0-freeze.md`](../../rfcs/0001-protocol-1.0-freeze.md).
2. `pnpm production-publish:gate -- --write` then `pnpm production-publish:gate` — must emit `unlocked: true`.
3. Set `NPM_ALLOW_PROD_PUBLISH=true` and `PYPI_ALLOW_PROD_PUBLISH=true` in repository variables.
4. Tag `v1.0.0`. `release.yml` loads `FREEZE_RFC_UNLOCKED` from `rfcs/appendix/production-publish-gate.json` before choosing production registries.

Validate configuration locally:

`pnpm publish:provenance`
`pnpm production-publish:gate`

## Post-pack integrity verification

After pack, the release workflow records SHA-256 digests for every publishable tarball and verifies them again immediately before upload. A digest mismatch aborts the release.

Commands (repo root):

1. `pnpm publish:integrity:record` — pack and write `artifacts/release-pack-integrity/manifest.json`
2. `pnpm publish:integrity:verify` — re-pack and assert digests still match the manifest
3. `pnpm publish:integrity:prove` — seeded red→green proof for digest drift detection

The release workflow runs record → verify around publish on every tag push and PR rehearsal path.

## CycloneDX SBOM (release assets)

Every tagged release generates CycloneDX 1.5 JSON SBOMs and attaches them to the GitHub Release:

1. `pnpm sbom:generate` — writes `artifacts/sbom/npm-workspace.cdx.json` and `artifacts/sbom/sutra-sdk-python.cdx.json`
2. `pnpm sbom:check` — asserts release.yml still generates, uploads, and attaches SBOMs

See [`security/SBOM/README.md`](../../security/SBOM/README.md). The release workflow is the only path that attaches SBOMs to GitHub Release assets — never from a laptop.

## Artifact signing verification (pre-production upload)

Unsigned artifacts **cannot** reach production registries. Before production npm/PyPI upload, the release workflow runs:

```bash
pnpm signing:verify -- --mode=policy          # always — workflow + checklist wiring
pnpm signing:verify -- --mode=production      # when PROVENANCE_ENABLED or PyPI prod
```

Production mode asserts:

1. **npm provenance attestation** — `PROVENANCE_ENABLED=true` and CI/OIDC only (`NPM_CONFIG_PROVENANCE`); laptop/local production publish is refused
2. **Wheel / tarball digests** — `artifacts/release-pack-integrity/manifest.json` SHA-256 entries exist (recorded by `pnpm publish:integrity:record`)
3. **PyPI production** — CI only; TestPyPI rehearsals do not require production signing

Verify commands for operators (repo root):

| Command | What it checks |
|---------|----------------|
| `pnpm publish:provenance` | release.yml OIDC + provenance config |
| `pnpm publish:integrity:verify` | tarball digests match the recorded manifest |
| `pnpm signing:verify` | policy + (with `--mode=production`) blocks unsigned prod upload |
